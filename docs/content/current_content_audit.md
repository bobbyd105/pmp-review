# Current Content and Repository Audit

## Executive conclusion

The repository currently functions as a **hybrid study platform with a lightweight exam-simulation mode**.

- It is a study platform because it includes a browsable bank, lessons, progress dashboard, prompt library, and local authoring validator.
- It simulates one narrow exam behavior because quizzes present random, non-repeating questions without immediate feedback and reveal explanations at the end.
- It is not a full exam simulator: there is no timed exam mode, blueprint-balanced assembly, flag/review navigation, difficulty model, domain/task selection, or psychometric calibration.
- It is not yet a comprehensive course: the 26 lessons are ECO-task overviews, not a sequenced concept curriculum with objectives and prerequisites.

The architecture is deliberately simple and internally coherent, but content metadata is too coarse for adaptive learning, spaced repetition, concept-level diagnostics, or source traceability.

## Question model

### Current schema

Each record in `data/questions.json` contains:

- `id`
- `eco_domain`
- `eco_task`
- `question`
- `options[]`
- `correct_answer` as an exact option string
- `explanation`

### Strengths

- Simple, readable, local, and fully consumed by the current UI.
- IDs are unique; all 384 records have four unique options and a matching correct answer.
- All three ECO domains and all 26 current task labels are represented.
- Prompts and explanations are substantive by length: median prompt length is 41 words; median explanation length is 79 words.
- No exact prompt/explanation duplicates were found, and no prompt pair crossed a 0.72 token-Jaccard near-duplicate threshold.

### Limitations and defects

- No `difficulty`, concept/objective tags, PMBOK mappings, approach, source, cognitive level, version, review status, or quality-review metadata.
- `eco_task` is only checked as a non-empty string. Tests do not validate the canonical 26 domain/task combinations.
- Correct-answer position is severely biased: 371 of 384 answers are option index 1 (option B); only 3 are A, 10 are C, and none are D.
- Correct-answer length is a second cue: the correct option is the longest option in 349 of 384 questions.
- Scenario endings are repetitive: 212 prompts end with the equivalent of “What should the project manager do?” and another 18 ask what to do first.
- Difficulty cannot be measured from the present records. Uniform four-option shape and long scenarios do not establish exam-level difficulty.
- Concept breadth is uneven despite balanced ECO counts. The bank has little or no direct coverage of the PMBOK structural model, quantitative risk, stakeholder-analysis models, several formulas, named agile frameworks, AI responsibility, programs/portfolios/operations, and PMO/organizational structures.
- All question-to-lesson relationships are inferred through one broad ECO task. There is no direct concept-level mapping.

## Lesson model

### Current schema

Each record in `data/lessons.json` contains:

- `id`
- `eco_domain`
- `eco_task`
- `title`
- `body` as markdown-capable plain text
- optional `related_question_ids[]`

### Strengths

- Exactly one overview lesson exists for every current ECO task.
- All 384 questions are linked from a lesson, and every link points to a question with the exact same `eco_domain` and `eco_task`.
- The minimal renderer emits React elements and does not inject raw HTML.
- The lesson bodies are concise review notes aligned to common PMP judgment patterns.

### Limitations

- Lessons are short overview assets: median body length is 163 words (range 133–303).
- None has explicit learning objectives or prerequisite metadata.
- There is no module, sequence, lesson type, concept taxonomy, PMBOK mapping, approach, glossary linkage, formula linkage, source traceability, version, review state, or last-reviewed date.
- `related_question_ids` is regenerated by ECO task, so it proves task alignment but not that a question tests the lesson’s specific concept.
- The current single lesson per task must not be expanded into an unstructured catch-all; doing so would make prerequisites, targeted review, and adaptive selection impossible.

## Prompt model

### Current schema and behavior

`data/prompts.json` stores eight records with `id`, `title`, `description`, and `prompt`. `PromptHelper` renders static text and copies it to the clipboard. No prompt is executed and no API is called.

### Assessment

- The architecture correctly keeps AI outside the app’s runtime and constitutionally preserves offline use.
- Prompt coverage is primarily study assistance and single-question authoring.
- There is no prompt metadata for purpose, input contract, output contract, target model, safety requirements, source expectations, version, or review history.
- Prompt p001 asks for explanations that compare each distractor, but the production question schema stores only one undifferentiated explanation.
- Prompt p008 requires the current minimal schema and cannot produce future traceability/difficulty metadata.
- No prompt currently supports lesson generation under a formal lesson contract, coverage-gap targeting, duplicate checking, or source-citation metadata.

## Relationships and data flow

```text
questions.json ──> Question Bank
       │          Quiz ──> localStorage session/history ──> Dashboard
       │
       └── ids referenced by lessons.json ──> Lessons

prompts.json ──> Prompt Helper ──> clipboard only

questions.json + lessons.json ──> Content Studio validation ──> clipboard snippet only
```

Relationships are one-way and read-only except browser-local quiz state. This is safe and understandable, but it lacks a canonical curriculum/objective layer between lessons and questions.

## Validation audit

### What is enforced

- Minimum record counts and domain breadth.
- Required question/lesson fields.
- Valid top-level ECO domain names.
- Unique IDs and cross-file question/lesson ID separation.
- Question option uniqueness and exact correct-answer membership.
- Lesson minimum body length and basic placeholder rejection.
- Referential integrity and uniqueness of lesson question IDs.
- Prompt shape, uniqueness, minimum length, and no embedded URLs.
- Content Studio mirrors most current data-contract rules for one pasted record.

### What is not enforced

- Canonical ECO task/domain pairs.
- Answer-position balance or option-length cueing.
- Duplicate/near-duplicate concepts and scenarios.
- Difficulty, cognitive level, approach, PMBOK, source, or review metadata.
- Bidirectional concept relationships.
- Coverage dataset consistency because no dataset existed before this engagement.
- Content accuracy; all current tests are structural.

### Drift risk

Content Studio duplicates contract logic from tests. The duplication is documented and partially protected by tests against real data, but a future metadata migration would require coordinated updates. A shared pure schema/validator module should be considered only when a real migration begins.

## UI audit

### Strengths

- Small, explicit components with static imports and clear data ownership.
- Accessible labels, headings, radio groups, status/alert regions, and semantic tables/lists.
- No hidden writes: authoring and prompt tools state their clipboard-only behavior.
- Quiz feedback timing is consistent with the recorded decision.

### Constraints

- Seven views after the coverage screen will make the flat button row crowded; the new screen should reuse the pattern for this limited slice, while navigation grouping remains a follow-up decision.
- Browse and Lessons render every record with no filters or virtualization.
- The quiz cannot target weak domains/tasks or honor a curriculum sequence.
- The Dashboard aggregates only ECO domain accuracy, not task, concept, difficulty, recency, or spaced-repetition state.

## Test architecture audit

Tests cover pure quiz/history/validator logic and each component’s primary behavior. The suite is appropriately local and deterministic. Full-bank render tests have required an increased timeout as content grew.

Material gaps for the new architecture are:

- no content-coverage data contract;
- no validation of source/ECO/PMBOK mappings;
- no coverage-screen tests;
- no bank-quality tests for answer-position or answer-length bias;
- no tests for concept/objective traceability.

This engagement will add coverage-data and screen tests only. Bank-quality defects will be reported rather than fixed because production questions are out of scope.

## Documentation audit

- `docs/app-map.html` is the strongest orientation artifact and was broadly accurate before this slice.
- `docs/progress.md` has accurate counts but is difficult to scan because every content batch is retained in the main current-state document.
- `docs/content_plan.md` correctly reports ECO breadth but equates task counts with curriculum coverage; it needs a pointer to the concept-level model and coverage matrix.
- `README.md` is not adequate onboarding documentation.
- The two local curriculum planning files are useful but ignored and therefore cannot serve as durable repository architecture.
